Pipeline reference

Telemetry → wear‑rate multiplier → cost‑per‑pick.

This page documents the data and modeling pipeline. Raw cobot telemetry is sliced into 2-second windows, aggregated, normalized against the robot datasheet, fed to a regression model that predicts a wear-rate multiplier, and then plugged into a deterministic cost-per-pick formula.

02 · records

Record schema

The NIST UR5 dataset ships as CSV files with exactly 73 columns. Each row is a snapshot of the robot state at one timestamp. Sample rate is 125 Hz, so rows arrive every ~8 ms. A typical run file holds ~8 600 rows, i.e. about 70 seconds of motion.

Of the 73 columns, the pipeline aggregates 36 columns grouped into six families:

  • Joint current — six columns actual_current_J1…J6 in amps. Primary load proxy: high current ⇒ high motor torque demand.
  • Joint velocity — six columns actual_velocity_J1…J6 in rad/s. Angular speed of each axis.
  • Joint temperature — six columns joint_temperature_J1…J6 in °C. Thermal aging of bearings and lubricant.
  • Target torque — six columns target_torque_J1…J6 in Nm. What the controller demanded from each joint.
  • TCP force — six columns at the tool center point: force_x, force_y, force_z, rx, ry, rz. Reaction force at the gripper.
  • Tracking error — six derived columns per joint: target_position − actual_position in radians. How tightly the robot follows the planned path.

Joint position is not aggregated directly — absolute angles carry no wear signal. The signal lives in the delta between target and actual position, i.e. the tracking error.

The remaining 37 columns (target position, target velocity, target current, acceleration, control current, cartesian pose) are loaded but not aggregated by the current pipeline. They are reserved for future anomaly detection.

03 · windowing

Stream → fixed-size samples

A single 8 ms snapshot is too noisy to model. Rows are grouped into non-overlapping 2-second windows. At 125 Hz each window contains roughly 250 rows. The trailing partial window at the end of a run is discarded.

For each of the 36 selected columns the pipeline computes seven aggregates inside the window:

vCntsample count
vFreqeffective sample rate
vMaxmaximum
vMinminimum
vStdstandard deviation
vTrendlinear slope
valuemean

36 columns × 7 aggregates = 252 values per window. They are flattened into a row keyed <col>_<stat> — e.g. actual_current_J1_vMax. A window therefore becomes one row in a 252-column table; this is the immediate pre-model artifact.

The aggregate set follows the PHM Society 2021 SCARA convention. Same stat schema across robot types so the downstream model can transfer.

04 · features

252 → 10 dimensionless + 2 categorical

The 252-aggregate vector is not fed to the model directly. It is reduced to ten dimensionless numeric features and two categorical features. Each numeric feature is divided by a datasheet constant so the model sees relative load instead of absolute values.

Dividing by rated values makes the feature space robot-agnostic: a UR5 at 150 % rated current and a SCARA at 150 % rated current land in the same region of feature space, even though their absolute amperages differ.

05 · labels

Wear-rate label generation

NIST UR5 ships without explicit failure labels — no sensor reports „this window degrades 1.7× faster than nominal". Labels are constructed from three assumptions grounded in materials science:

  1. Load stress (Basquin's law). Fatigue damage scales as a power of stress amplitude. We take the peak joint current in the window (proportional to torque) and raise the ratio against rated current to α = 2.5. 50 % overload therefore implies roughly 2.75× wear rate, not 1.5×.
  2. Thermal aging (Arrhenius). Lubricant and bearing degradation grow exponentially with temperature. We multiply by exp(0.05 · ΔT) where ΔT is degrees above the reference temperature (30 °C). At 60 °C the thermal factor is e1.5 ≈ 4.5.
  3. Cycle density. More cycles per unit time means less cooling and recovery. We multiply by rated_cycle_time / observed_cycle_time. In the NIST data this ratio is taken from the filename tag (fullspeed = 1.0× rated, halfspeed = 2.0× rated). True per-pick detection is deferred.

The three factors combine multiplicatively:

raw_wear = (I/Irated)2.5 · exp(0.05·(T − 30°C)) · (Trated / Tobs)

Anchor normalization. The raw value has no meaningful scale on its own. The pipeline takes the median over all windows that match warm + fullspeed + light + train and pins that median to 1.0×. Every other window is then expressed relative to it. A window with a label of 2.0× wears twice as fast as the nominal-load reference.

α = 2.5 and k = 0.05 are literature defaults for steel/lubricant systems. They are a first-order approximation; in production they would be tuned via cross-validation and eventually fit against field-data RUL outcomes.

06 · model

LightGBM regression on log(wear)

The model is a gradient-boosted-tree regressor (LightGBM). It is trained on the logarithm of the multiplier rather than the multiplier itself, because multipliers are multiplicatively distributed — 0.5× and 2.0× are symmetric around 1.0× on a log scale.

  • Input: 10 dimensionless numeric features + 2 categorical (integer-encoded).
  • Hyperparameters: num_leaves = 15, learning_rate = 0.05, n_estimators = 300, early_stopping = 30.
  • Loss: RMSE on log(wear_rate_multiplier).
  • Inference: exp(prediction), then clipped to [0.3, 5.0].
  • Persistence: booster serialized to artifacts/wear_rate_lgbm.txt.

GBDT was chosen over a neural net because the feature count is small (10 numeric), the training set is modest (thousands of windows), and tree boosting gives free SHAP-based feature importance for downstream interpretability.

07 · cost

Cost-per-pick · four components

Once the model emits a wear_rate_multiplier per pick, the cost-per-pick calculation is pure arithmetic — no second model. Four components, all derivable from the datasheet plus a finance/operating config:

energy
power_w · load · cycle_t · €/kWh

Per-pick electricity: rated power times load factor times cycle duration times electricity price. UR5 at 150 W, 2 s cycle, nominal load: ≈ €0.000025. The smallest component.

wear
(cost_new / lifetime) · multiplier

Capex amortized across lifetime picks, scaled by the live multiplier. UR5 = €35 000 / 30 M picks = €0.00117 at 1.0×; at 1.5× it becomes €0.00175. The only component that reacts to telemetry.

capital
cost_new · interest_rate / lifetime

Cost of capital tied up in the asset, amortized over lifetime picks. UR5 at 5 %: €35 000 · 0.05 / 30 M = €0.00006.

maintenance
cost_new · pct_p.a. / picks_p.a.

Annual maintenance as a fraction of capex, divided by annual picks. UR5 at 5 % p.a. and 2 M picks/year = €0.000875.

Σ UR5 nominal (multiplier = 1.0) ≈ €0.0020 per pick
Σ UR5 under load (multiplier = 1.5) ≈ €0.0026 per pick

Assumptions. Linear depreciation across lifetime; flat electricity price and interest rate; maintenance modeled as an annual percentage of capex; energy taken from datasheet rated power, not from real-time current readings. The wear-rate multiplier is the only term that varies with telemetry; everything else is static config.

08 · ucs

UCS drop-in · zero-shot transfer

Because all numeric features are dimensionless ratios against datasheet constants, the same trained model can score telemetry from any robot whose schema can be mapped to UCS. The UCS agent automates that mapping with two structured LLM function calls:

  1. User uploads two files — the foreign telemetry CSV and the foreign datasheet.
  2. Agent emits two function calls: one maps the column names onto the UCS schema, the other extracts datasheet constants (rated current, rated cycle time, rated payload).
  3. User confirms or edits the mapping in the UI.
  4. Normalizer applies the extracted constants to compute UCS features.
  5. Inference runs against the UR5-trained model — no re-training.
  6. Cost engine emits €/pick using the foreign datasheet's capex and lifetime.
